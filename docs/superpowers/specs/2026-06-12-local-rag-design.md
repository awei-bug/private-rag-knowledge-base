# 本地 RAG 知识库问答系统轻量化设计

## 目标

将当前偏企业控制台形态的 RAG 项目收敛为面向单机使用的本地知识库问答系统，保留网页界面，但页面只聚焦两件事：

- 供：导入知识
- 求：基于已导入知识发起问答

本次改造以复用现有后端主链路为优先，不从零重写检索与问答能力。

## 用户场景

目标用户是在本机上整理资料并直接提问的人，不需要多用户协作、权限审批、审计追踪或后台运维视角。

典型流程：

1. 打开网页。
2. 通过上传文件或同步本地目录导入知识。
3. 在同一页面输入问题。
4. 查看回答和引用来源。

## 范围

### 本次纳入

- 保留 FastAPI 后端与 React 前端。
- 保留文件上传导入。
- 保留本地目录同步导入。
- 保留文档列表和基础删除能力。
- 保留问答能力与引用片段展示。
- 同时支持本地默认模式和外部 OpenAI 兼容 API 模式。
- 将前端重构为单页极简界面。

### 本次移出主产品流

- 登录页
- RBAC 权限控制
- 审计日志页面
- 查询日志导出
- 检索调试页
- 控制台式导航和系统总览

说明：相关代码可以暂时保留在仓库中，但不应再成为默认交互路径，也不应在首页暴露。

## 方案选型

### 方案 A：只删页面，后端基本不动

优点：改动最小。
缺点：产品形态仍然残留控制台心智，接口和状态管理也会继续受认证体系牵制。

### 方案 B：重做前端单页，后端保留主链路并弱化企业能力

优点：交付速度和最终形态平衡最好，能最大化复用导入、检索、问答链路。
缺点：后端内部仍会存在部分历史结构，需要做一次接口和依赖收敛。

### 方案 C：前后端全面重构为全新轻量项目

优点：概念纯。
缺点：代价大，容易破坏现有可用能力，不适合作为首轮改造。

结论：采用方案 B。

## 系统设计

### 前端

前端改为单页应用，不再保留多页面导航。页面分为两个主区块：

- 供：知识导入区
- 求：知识问答区

#### 供：知识导入区

包含以下交互：

- 文件上传
- 本地目录路径输入与同步按钮
- 已导入文档简表
- 文档删除入口
- 最近一次导入结果提示

该区域只展示导入所需的最少信息，不显示 ACL、用户角色、元数据调试项等企业字段。

#### 求：知识问答区

包含以下交互：

- 问题输入框
- 提问按钮
- 回答文本
- 引用来源列表
- 当前运行模式提示

引用来源列表至少展示：

- 文档标题或文件名
- 片段内容
- 相似度或排序分数（如现有接口容易保留）

### 后端

后端继续保留现有导入、检索、问答服务实现，但对外接口改为本地版默认体验。

建议默认对外保留的能力：

- `POST /api/v1/documents/upload`
- `POST /api/v1/documents/sync/local-dir`
- `GET /api/v1/documents`
- `DELETE /api/v1/documents/{document_id}`
- `POST /api/v1/query`
- `GET /health`
- `GET /config`

需要弱化或从前端调用链中移除的接口：

- 认证相关接口
- 查询日志接口
- 检索调试接口
- 文档详情和分块详情接口（可保留但前端不默认展示）

### 认证与访问控制

本地版默认应取消登录门槛，页面打开即可使用。后端可采用以下任一轻量方式：

- 彻底绕过当前认证依赖，接口默认使用一个固定本地用户上下文
- 或新增本地模式开关，在本地模式下跳过认证校验

推荐第二种。这样既不破坏现有结构，也保留未来切回鉴权模式的空间。

### 运行模式

系统支持两类运行方式：

- 本地模式：使用当前本地 embedding 与本地模板生成能力，开箱即用
- API 模式：使用 OpenAI 兼容接口进行 embedding 和回答生成

默认行为：

- 如果未配置外部 API，则系统使用本地模式
- 如果配置了兼容 API 所需参数，则系统可切换到 API 模式

前端不做复杂设置中心，只展示当前模式和必要提示。模式切换优先通过 `.env` 管理，而不是页面内高级配置。

## 数据流

### 导入流

1. 用户上传文件或输入本地目录。
2. 前端调用导入接口。
3. 后端解析文档、切块、生成向量并入库。
4. 前端刷新文档列表并展示结果提示。

### 问答流

1. 用户输入问题。
2. 前端调用问答接口。
3. 后端执行检索、重排、回答生成。
4. 前端展示回答和引用来源。

## 错误处理

前端需要覆盖以下主要错误场景：

- 上传空文件
- 本地目录不存在或不是目录
- 未导入任何文档时发起提问
- 外部 API 配置不完整
- 后端不可达

错误提示应直接面向普通使用者，不出现内部控制台术语。

## 测试与验证

本次改造完成后至少验证以下内容：

1. 前端可以直接打开，不需要登录。
2. 上传单个文件后，文档列表会更新。
3. 同步本地目录后，文档列表会更新。
4. 文档删除成功后，列表会刷新。
5. 提问后能返回回答与引用来源。
6. 未配置外部 API 时，本地模式可运行。
7. 配置兼容 API 后，接口仍可正常回答。

## 实施顺序

1. 精简前端状态管理，移除登录和控制台导航依赖。
2. 重写前端为单页“供 / 求”布局。
3. 后端新增或接入本地模式免登录机制。
4. 清理前端 API 调用，只保留本地版主链路。
5. 更新 README 和默认配置说明。
6. 运行后端测试与前端构建验证。

## 非目标

以下内容不属于本轮改造目标：

- 多用户系统
- 企业权限体系
- 审计报表
- 后台管理台
- 在线模型参数调优界面
- 全量后端架构重写

## 风险与约束

- 当前仓库不是 git 工作树，无法按常规流程提交设计文档版本。
- 现有前端中文文案存在编码异常，改造时应顺手统一为正常 UTF-8 文案。
- 如果认证逻辑在后端依赖链中耦合较深，首轮应优先增加“本地模式旁路”，而不是大面积删除认证代码。
